# routines

The routines module gives a campaign its recurring upkeep without a human remembering to run anything. It defines a maintenance routine as a contract rather than a how-to, states a runner preference chain for hosting that contract, and ships the freshness loop as the first standard routine together with one worked adapter. It touches [publish](publish.md) and [retrieval](retrieval.md), depends on [cli](cli.md), and is documented at version 0.1.0.

## The routine contract

A routine is a contract with three obligations, and any runner that honors them is a valid adapter. It must be idempotent, so running twice equals running once. It must be safe to miss, so a skipped run degrades freshness but never correctness. And it must be safe to double-run, so two overlapping runs cannot corrupt anything. Because the contract is stated in these terms rather than as vendor tutorials, the maintaining agent maps it onto whatever scheduler the owner's world already offers. A routine automates verbs that already work; the composed verbs are vendored and green by hand first, since a routine never debugs what it schedules. Refusals are loud, never silent: a routine that hits a lint error or a firewall refusal stops, surfaces the reporter's output through the runner's notification surface, and fixes nothing — judgment stays with the owner and their agent.

## Runner preference chain

The runner is chosen by walking a fixed preference chain top-down and taking the first rung the owner's world already has. The rungs are hosted agent routines, which are the best value when the owner's agent subscription includes them; GitHub Actions, whose free tier fits cloud-mode campaigns whose repository lives on GitHub; and local cron-esque schedulers such as launchd, Task Scheduler, or cron, which fit a machine that is reliably awake. The chain exists so the free rung is always reachable, and its governing rule is that no new paid service is introduced merely to host a routine. This is the module's cost posture: the freshness routine is deterministic and spends zero model tokens, so the free rung always suffices, while a routine that reads transcripts or lints semantically must document its cost posture and pre-compress its inputs before any model sees them.

## The freshness routine

Freshness is the first standard routine. Its purpose is that every player surface — the published site, the retrieval worker, and the [lore bot](lore-bot.md) corpus — tracks the wiki without human hands. The lore bot already self-refreshes by polling, so the routine covers the surfaces that need a push: the projection, the site, and the worker corpora. It composes six verbs in a fixed order and stops on the first failure. First `eddic lint --strict`, where errors or warnings both block, because a routine must never publish what a human would not. Then `eddic project`, which refuses all-or-nothing on firewall breaches, followed by `eddic build`, then a Pages deploy of the built site. Then `eddic stage` prepares the worker corpora, and finally a worker deploy from the campaign's `worker/` directory. Idempotency holds because every step is a pure function of the wiki tree, so an unchanged wiki republishes identical content. Missing a run leaves surfaces stale but never wrong, since the last published state was itself lint-clean and firewall-checked. Overlapping runs converge: the derived directories are rewritten atomically enough, and the deploys are last-write-wins of identical content. The default cadence is event-driven on wiki change, with a polling fallback every thirty minutes.

## The GitHub Actions adapter

The module ships one worked adapter that implements the freshness contract for a cloud-mode campaign whose repository lives on GitHub. It is a workflow that triggers on pushes touching the `wiki/` or `contribs/` trees, so its cadence is event-driven and no schedule is needed. Its job runs the six contract verbs in order — strict lint first, then project, build, site deploy, stage, and worker deploy — with no continue-on-error anywhere, so any refusal stops the chain and shows the owner a failed check. Adopting it requires only filling the Pages project name and setting the two Cloudflare secrets it names. For the hosted agent and cron-esque rungs there is no shipped template; the contract file is the specification, and the maintaining agent implements it natively against the host's own machinery.

## Applying the pattern

The procedure picks a routine's contract, maps it onto the chosen runner, and logs the result. When adopting freshness on GitHub Actions the adapter is a working start; on the other rungs the contract file is implemented directly. Every applied routine is logged as a `schema` entry naming the routine, its runner, and its cadence, because routines are part of the campaign's shape. Verification is twofold. The deterministic check parses the Actions template as YAML and confirms it composes only the contract's vendored verbs in order with strict lint first and refuse-on-error semantics, and that the contract file states all five required sections. The live check triggers the runner once by hand and watches the whole chain succeed, then triggers it twice back-to-back to observe the double-run clause resolve to a clean no-op or a clean repeat.

## Related
