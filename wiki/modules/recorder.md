# recorder

The recorder module gives a campaign its own session-recording Discord bot: a bot distinct from the [lore bot](lore-bot.md), with a different lifecycle. Where the archivist is always-on and cloud-cheap, the recorder is session-time-only, voice-heavy, and wants to run beside the disk that the transcriber reads. It captures per-speaker audio in a Discord voice channel and stages it, one track per consenting member, straight into the transcriber's on-disk layout on the DM's machine. Consent is structural rather than a matter of policy: a member's microphone is captured only after they react to a consent post in the voice channel's text chat, and that gate is enforced inside the audio sink, so unconsented packets are dropped before any file exists.

## Consent gate

At session start the bot posts a consent message in the channel and reacts to it with the studio-microphone emoji. Only members who add that same reaction are recorded; an unreacted microphone is never written, and removing the reaction stops capture from that moment forward. The gate lives in the sink's write path. The sink is called once per decoded audio frame with a frame and a user; a frame whose speaker is not yet attributed is counted as unattributed and discarded, a frame from a member absent from the consented set is counted as unconsented and discarded, and only a frame from a consented member is appended to that member's track. Because the check precedes the first write, a non-consenting member produces no file at all, not merely an empty one. Emoji comparison normalizes the Unicode variation selector, since reactions arrive both with and without it. Reaction events are read as raw payloads, so the bot never depends on message-content or member-cache state to know who consented. The default posture is strict per-session reaction: consent is a fresh tap each session, with no standing memory carried across sessions. Voices audible through another member's open microphone remain present on that member's track, as in any recording, and the consent post says so.

## Transparency

Recording announces itself three ways at start. The consent post itself is the authoritative signal and states who is currently being recorded, updating live as reactions arrive and are removed. An audible two-note rising chime, synthesized as forty-eight-kilohertz stereo PCM, plays into the channel. Unless declined for the session, a visible recording status is set on the voice channel via a REST call, since py-cord has no wrapper for voice-channel status yet; this is best-effort and never breaks recording if it fails, because the consent post remains the source of truth. Silent capture never occurs.

## DAVE voice receive

Discord's DAVE end-to-end encryption broke voice receive across the Python ecosystem in March 2026. The recorder works anyway. A [companion](companion.md) module, imported at start-up before any voice connection is created, monkey-patches py-cord 2.8.0's voice engine at import time so its receive path decrypts DAVE frames correctly, wires the voice client into the sink, and hardens the packet-router threads against stop-time noise and undecryptable frames. The fixes are adopted from py-cord's in-progress voice-receive rework and target py-cord 2.8.0 exactly; nothing in site-packages is edited. When upstream lands working voice receive, this patch layer retires. Live capture through transcription was proven end to end on 2026-07-18. Because of this dependency the pinned runtime is py-cord 2.8.0 with its voice extra plus the DAVE bindings, and the versions must not drift.

## Setup and running

The recorder is a second Discord application, created through the same portal flow the lore bot uses, with its token reset once into the recorder's own gitignored variables file. It needs no privileged intents at all: slash-command interactions, consent reactions as raw reaction events, and voice states all arrive under default intents. Its invite requests the bot and application-commands scopes with permission to view channels, send messages, read message history, add reactions, connect, speak, and set voice-channel status. A bot invited with application-commands but no bot scope is authorized yet absent from the member list and sees zero guilds, which is a common failure mode; re-inviting with the bot scope fixes it.

Templates for the recorder and its DAVE patch are copied into a per-campaign bot directory alongside a five-line entry point that constructs a py-cord bot and hands it to the module's setup call. The application registered as a campaign service so the owner never types the runtime incantation: the CLI dispatcher supplies the pinned runtime and launches the bot in the foreground, which by construction means one running copy, since a Discord-side ghost that outlives a hard kill is exactly what a single-process foreground launcher prevents. During a session the owner starts recording with a slash command in their current voice channel, and stops it with another. The session directory is named by its exact start timestamp rather than by date alone, so a session that crosses midnight, two sessions sharing a day, and a crash-and-restart never collide or squash existing tracks. On stop, the tracks are staged, the consent post is edited to note that recording ended, and a witness line is appended to the campaign's operation log recording how many tracks landed and where. See the [module contract](../concepts/the-module-contract.md) for how services, manifests, and pinned runtimes fit together, and [the deterministic core](../concepts/deterministic-core.md) for why the consent gate is code rather than agent judgment.

## Destination and lifecycle

By default recordings land in the campaign's raw-sessions directory on the DM's machine, which is the transcriber's own layout and requires no hosting. Per-speaker tracks are written as well-formed forty-eight-kilohertz sixteen-bit stereo WAV files named by track order and the member's display name. A cloud host with object storage is a deliberately unbuilt future mode, revisited only if a table's DM machine genuinely cannot host sessions. The bot is not a resident service; it is launched for a session and stopped afterward, because nothing needs it between sessions. Transcription of the staged tracks is a separate, deliberate maintenance step.

## Relation to other modules

The recorder assumes the [capture](capture.md) pattern has been applied, since capture chooses the campaign's recording route and keeps a free third-party recorder as its own default; the recorder is the self-hosted, consent-gated alternative for tables that want per-speaker tracks under their own control. It stages directly into the [transcriber](transcriber.md)'s layout, is launched through the [cli](cli.md) service dispatcher, and stands beside the campaign's other Discord application without sharing its lifecycle.

## Verify

The module's verifier runs without Discord or the DAVE bindings installed. It compiles both templates, then exercises the consent core against a stubbed library: emoji normalization accepts reactions with and without the variation selector, the sink drops and counts unattributed and unconsented packets without creating a file, consented audio lands as a well-formed per-speaker WAV under the member's display name, and revoking consent mid-stream closes the gate. A live two-member test is run once per setup to confirm that a consenting member's audio is present and transcribable while a non-consenting member is absent from every track.
