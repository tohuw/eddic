# streamdeck

The streamdeck module gives the owner an Elgato Stream Deck control suite for the [recorder](recorder.md) bot: one key to start a recording, one to stop, a key that reports status, and a help key, with optional table-time keys for other campaign actions. It has two halves that ship together — a control surface inside the recorder, and a button pack the owner binds — because a Stream Deck key cannot speak Discord's slash-command protocol, but it can run a script, and a script can curl a local endpoint.

## The control surface

The recorder bot exposes a small HTTP control surface bound to the loopback address inside its already-local process. Its endpoints start recording, stop it, and report status, plus an unauthenticated liveness check that a button can use to show whether the bot is up. Every endpoint calls the recorder's shared session core, the same code path the `/record` slash commands take, so a Stream Deck key and a slash command are two triggers for one action and can never diverge; in particular the consent post is posted publicly and the consent gate is enforced identically no matter which trigger started the recording. The surface is bound to `127.0.0.1` and is never exposed off the machine by any configuration. On a single-user machine an unauthenticated posture is defensible, but the owner can set a shared secret, sent as a header, to keep other local processes from driving the recorder. Because a control request carries no interaction context naming a channel, the surface resolves its target from an explicit channel or guild setting, from the voice channel the configured owner is in, or from the one populated voice channel, and refuses with a clear message when more than one channel is populated. The surface is on by default and is disabled with a single environment flag.

## The button pack

The module's generator stamps a folder into the campaign holding one script per key. Each script curls the matching control endpoint and is bound to a Stream Deck key through the app's built-in System-Open action, so no plugins are needed; the scripts are `.command` files on macOS and `.cmd` files on Windows, and they work equally from a double-click or any other launcher. Alongside the scripts the generator writes a README with the exact bind steps and an experimental `.streamDeckProfile`, a zipped manifest that maps keys to those scripts and can be imported as a starting point. Because Stream Deck profile formats drift between application versions and the module cannot test against real hardware, the profile is offered as best-effort convenience and the manual System-Open bind is documented as the reliable path. When the owner asks for extras, the generator adds optional keys that run campaign `eddic` verbs in a terminal — a suggestions inbox, a publish, a convene — each a thin wrapper kept only if the campaign actually provides that verb. The token, port, and base URL are baked into the scripts at generation time, so the pack is regenerated if the surface's posture changes, and because it can carry the control token it is kept out of version control.

## Setup and running

The recorder pattern must be applied and its bot running, with the control surface reachable — a liveness curl is the preflight. The owner sets the surface's posture in the recorder bot's variables file: whether it is enabled, its port, an optional shared secret, and an optional target-channel hint. The pack is then stamped for the owner's operating system, with the same secret the recorder uses and, if wanted, the extras flag; the generator writes the pack folder and prints the bind reminder. The owner drops a System-Open action on each key and points it at the corresponding script. The application is recorded in the campaign manifest so an upgrade can restamp it.

## Verify

The module's verifier runs with no Stream Deck hardware. It compiles the generator and golden-tests its pure builders: the core scripts curl the right method and endpoint and include the shared secret only when one is configured, the Windows target emits carriage-return line endings, a full pack materializes with the four core keys, an executable bit, a README documenting the System-Open bind, and a `.streamDeckProfile` that is a valid zip whose manifest maps the first key to a System-Open on the start script, extras appear only when requested, and building for both operating systems stamps both script kinds. The recorder module's own verifier proves the control router's authentication and dispatch. Live, with the recorder running, the endpoints are curled directly to confirm that a control start posts the public consent message and begins recording exactly as the slash command would, and then a bound key is tapped to confirm the same.
