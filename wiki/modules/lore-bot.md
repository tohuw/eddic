# lore-bot

The lore-bot module puts a lore-keeper in the table's Discord. Mentioned with a question — "what the hell is a Celeste" — it answers from the campaign [wiki](wiki.md), only from the wiki, and cites the pages a reader can rabbithole into. Its corpus is the player projection by construction, so it cannot leak what the projection does not contain; the module depends on [cli](cli.md) and wiki, and touches only a `bot/` directory in the campaign. It reached 0.3.0 with the Anthropic path verified live end-to-end and the OpenAI path golden-tested but not yet deployed.

## What it is

One process, no inbound ports, that connects to Discord as a bot application and answers when it is @mentioned or when a message lands in a configured auto-answer channel. Every answer is generated by a model that has been handed the corpus as a cached system block plus a persona and, optionally, a private roster. The bot holds the corpus in memory and refreshes it on a timer, so a published wiki change becomes an answerable fact without a restart.

## The corpus is the projection

The corpus is built by concatenating every content page under the projection directory, each page headed by its relative path in a `=== path ===` marker so the model can cite exactly what it read. Non-content files — `CLAUDE.md`, `AGENTS.md`, `README.md`, `log.md` — are excluded. Because the input is the player projection and never the DM master, the bot inherits [the firewall](../concepts/the-firewall.md) rather than re-implementing it: material that never made it into the projection is simply not in the model's context. This is the module's central safety property, and it is structural, not a matter of the model's discretion. Ask about a fact that lives only on a DM page and the correct answer is a plain "the archive doesn't say."

## How it stays fresh

Freshness is a poll against the corpus source, honoring the module's freshness contract rather than any push mechanism. In local mode — the default when a machine is reliably awake during play — the source is the projection directory on disk, and the poll compares a fingerprint (a hash of every content file's path, size, and modification time) against the last load; a difference triggers a rebuild. In cloud mode the source is the campaign repository: the poll fetches the branch HEAD SHA, and on a change refetches the repository tarball through the API and rebuilds the corpus from the configured subdirectory, needing no git on the host. Either way the refresh interval is `REFRESH_MINUTES`. The owner-only `!lore status` and `!lore reload` commands are an escape hatch for inspecting or forcing a reload, never the mechanism itself.

## Persona and the hard rules

The persona is a plain markdown file the campaign owns. Its default is a courteous, dry-witted chronicler, and the owner changes the bot's character simply by asking for a new voice — the persona is authored prose, not a filled-in template. Whatever the character, a fixed set of hard rules rides underneath it and never yields to personality: answer only from the corpus, cite real pages, invent nothing, never speculate about where the story is going, and keep answers to at most two short paragraphs before citations. When `SITE_URL` is set, bot.py appends a citation instruction and a worked example to the persona at startup, binding the live site root beside the rule that uses it; when the site is unpublished the bot names pages in plain words rather than emitting a link that goes nowhere. The persona file is the owner's voice and nothing else — config-derived facts are appended at runtime, so plumbing changes touch the config, not the prose.

## Provider seam and prompt caching

The answer API is chosen by `PROVIDER`. The default and only live-verified path is Anthropic, which places the corpus in a system block with an explicit prompt-cache breakpoint and puts the persona and roster after it; caching the large, stable corpus prefix is what keeps the per-question cost trivial. Selecting OpenAI swaps in the Responses adapter, which leads its stable instruction prefix with the same corpus-first discipline and relies on automatic prefix caching. Both adapters keep the roster behind the cached region, so an optional roster file — the only place real player names may appear — is injected after the breakpoint and never enters the corpus, the repository, or the wiki. Each adapter's request shape is pinned by golden tests in the module's verify run, so a refactor cannot silently change what either API receives.

## Optional capabilities

The always-on bot is an extension point. If a capability module such as [convene](convene.md) has vendored its file beside bot.py, the bot imports it at startup and wires it to the same Discord client, gaining session-lifecycle behavior — scheduled events, quorum, recap announcements — off the same corpus and the same refresh cycle. A capability that fails to load or errors is logged and skipped; it can never break the bot's core Q&A. This is [the capability seam](../concepts/the-capability-seam.md) in practice: the pure corpus helpers the bot exposes are exactly what a capability builds on, without re-reading disk.

## Verify

The module's verify run unit-tests the pure helpers — config precedence with real environment beating the file, corpus build and non-content exclusion, fingerprint change detection, Discord message splitting, mention stripping, and the corpus page helpers — then compile-checks bot.py and both provider adapters without discord or anthropic installed, and finally runs the golden request-shape tests against fake SDKs. The live check, after setup, is a three-part ritual: ask a question answered by the wiki and expect a cited answer, ask one whose answer exists only on a DM page and expect the archive's blindness, then publish a change and confirm the bot reports it after the refresh interval.
