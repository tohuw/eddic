# The module contract

The module contract is the agreement every Eddic module keeps with the toolkit, the campaigns that adopt it, and the agents that apply it. A module is Eddic's shippable unit: a self-contained directory holding everything an agent needs to give a user one facility — a wiki, a lore bot, a published site, a maintenance routine. Modules are independently adoptable and arrive as community contributions by pull request, so the contract is what lets strangers' work merge without a central gatekeeper reading every line for correctness. It is deliberately not an API surface of many endpoints; submissions are agentic, so the contract is a set of good-faith semantics resting on a small deterministic floor that machines can check. The split mirrors the toolkit's founding principle: a [deterministic core](deterministic-core.md) that CI enforces, wrapped in an agent shell that a maintainer reviews.

## Anatomy of a module

A module lives at `modules/<name>/`. Two files are unconditionally required: `module.yaml`, the manifest declaring name, version, summary, what the module touches, its dependencies, and its cost posture; and `PATTERN.md`, the instructional layer written for an agent reader. Three directories are required exactly when the pattern references them and forbidden clutter when it does not: `scripts/`, the deterministic machinery run through the eddic [CLI](../modules/cli.md); `templates/`, the files a pattern stamps into a campaign; and `verify/`, the observable success checks. A pattern with no verify section does not merge — the floor requires one — but a pattern that never invokes a script needs no `scripts/` directory. Presence is contingent on reference, so a module carries no dead weight.

## The four-part pattern

Every `PATTERN.md` has four parts, in fixed order. Preflight comes first: deterministic checks that the environment can accept the pattern — dependencies present, configuration sane, any prior manifest state compatible. Preflight is where a mismatch fails cheap, before work begins. The procedure follows: prose directing the agent, invoking scripts for everything repeatable. The agent's judgment adapts the procedure to its environment but never replaces the scripts. Decision points are the only places the agent consults its user, and each one ships a recommended default, plus a when-it-is-worth-it heuristic where experience warrants. A decision point without a default is a contract violation, because defaults are what make "just do what you think is best" work mechanically — an agent told to proceed takes every default and asks nothing. Verify comes last: observable success criteria, not the empty assurance that it should work now, which the agent runs and reports faithfully. See [patterns, not code](patterns-not-code.md) for why the instructional layer is prose rather than an executable.

## The deterministic floor

The floor is the mechanical, CI-enforced half of the contract, small on purpose. Semantic review is bad at catching mechanical rot — a broken script reads fine — so the machine-checkable properties are pulled out and run on macOS and Windows runners. The manifest must be present and schema-valid. The pattern must carry all four parts, and every decision point must carry a default. Every script the pattern references must exist and execute in a smoke run. The verify section must exist and its checks must execute. No secrets or credentials may be committed. No symlinks and no bash-required machinery may appear, since deterministic work runs under the CLI so Windows is covered by construction. Finally, vendor claims must be metadata-backed: if a pattern names a vendor — a maintaining agent, answer client, or model provider — the manifest carries a dated `compatibility:` entry stating the role, the evidence state, and for a verified claim its evidence. Nothing below verified may be a decision point's default path, and because truth decays, entries are re-dated on re-test and demoted when a vendor moves. This extends to behavioral claims: a module promising how an answer client will conduct itself must verify that conduct adversarially, per client, before asserting it. The [lint](../modules/lint.md) module is the floor made concrete for wikis, and the [capability seam](the-capability-seam.md) is how the floor stays honest across runtimes.

## The semantic rubric

Above the floor is the rubric, reviewed in good faith by a maintainer and an agent reading with a critical eye. Anything repeatable should be a script, not prose asking an agent to improvise — deterministic where possible. A pattern must contain nothing a competent agent would produce unaided: contracts, invariants, proven procedure, and hard-won heuristics belong; vendor how-tos do not. This no-egg-sucking bar is also the anti-bloat rule, since content that fails it is a strictly worse pathway than what the agent would do anyway. And a module must keep Eddic's standards throughout — its vocabulary, principles, and tone; the firewall and authorship invariants respected; agent-agnostic, with no runtime-exclusive machinery lacking a stated fallback chain.

## Cost posture and versioning

The manifest declares a `cost_posture`: a free or local path, a paid or cloud path where one exists, and a heuristic for when the paid path is worth it. It never names dollar amounts, because prices rot and postures do not, and every paid recommendation names its free fallback. Every module must be satisfiable inside the baseline build unless the manifest says otherwise and the pattern says why. Modules version independently under semver. A campaign's manifest pins the version applied, and patterns must be idempotent — applying an already-applied pattern is a no-op — so an upgrade is a re-run of the pattern at a new version against the recorded state. See [releases and versioning](releases-and-versioning.md) for how versions move, and [contribs](../modules/contribs.md) for how contributed modules enter the tree.
