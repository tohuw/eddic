# Patterns, not code

Eddic delivers a facility to a campaign as a pattern an agent applies, not as an application a user installs and runs. The distinction is the project's organizing principle. An agent arriving to set up a campaign wiki, a lore bot, a published site, or transcription wants Eddic's facilities and not Eddic's source code; what it reads is an instructional layer that tells it how to stand the facility up in the user's own environment, invoking deterministic machinery for everything repeatable and exercising judgment only where the pattern marks a decision. The unit of delivery is therefore the pattern, and the shippable package that carries one is the [module](../modules/index.md).

## The pattern as the deliverable

A pattern is a module's instructional layer, written for an agent reader rather than a human operator. Every pattern has four parts in fixed order. Preflight runs deterministic checks that the environment can accept the pattern — dependencies present, config sane, prior state compatible — so failure is cheap and early. Procedure is prose that walks the agent through applying the facility, invoking the module's scripts for anything repeatable; the agent's judgment adapts the procedure to the environment but never replaces the scripts. Decision points are the only places the agent consults the user, each explicitly marked and each shipping a recommended default so that a user who says "just do what is best" can be served without a single question. Verify states observable success criteria the agent runs and reports faithfully, never "it should work now." This shape is what makes an agent-authored, agent-applied instruction a reliable install step instead of an improvisation.

## Why not code

The choice rests on two founding principles. The first is deterministic core, agent shell: scripts do everything repeatable, agents read patterns and make the judgment calls at marked decision points. An agent never does infrastructure's job — authentication, projection, cryptography, firewalls — and infrastructure never does judgment's. Handing the agent a runbook rather than a binary is how that separation is expressed at the seam where a facility meets a particular machine, since the same pattern must stand a wiki up on Windows or macOS, against whatever the local agent and its tools happen to be. The second is no egg-sucking: Eddic never teaches an agent what it already knows. A pattern states contracts, invariants, proven procedures, and hard-won heuristics; it never contains a vendor how-to that a competent agent would produce unaided. Content that fails this bar is deleted, because it is a strictly worse version of the pathway the agent would take anyway. Distributing code would violate both principles at once — it would freeze into one runtime what the agent should adapt, and it would carry the very boilerplate the agent does not need told.

## Application is recorded, not just performed

Because a pattern is applied rather than run, its application must leave an auditable trace. Each campaign keeps a manifest at `.eddic/manifest.json` recording which patterns were applied, at what version, with what parameters. This is what makes application idempotent — the same version with the same parameters applied twice is a no-op — and upgradable, since an upgrade re-runs the pattern at the new version against the manifest and the pattern states what changes on re-application. The manifest is the package manager for a model in which the install step is an agent following a runbook. The stable interface the patterns are written against is the [cli](../modules/cli.md): a contractual locus for deterministic workflows whose implementation can change beneath the patterns without any pattern being touched. Deterministic scripts such as the [lint](../modules/lint.md) checker are ground truth precisely because semantic review is poor at catching mechanical rot; the pattern points the agent at them and forbids improvised replacements.
