# AGENTS.md — {{SITE_NAME}} campaign wiki

This repository holds a campaign maintained under Eddic conventions:
raw sources, the DM's wiki (the single master), and generated
projections. This file is the schema — how the wiki is structured and
how you, the maintaining agent, work in it. Renderers and bots skip
this file, CLAUDE.md, and README.md; they are never published.

## Layers

- `sources/` — raw material: the DM's and players' own writing,
  shared documents, session transcripts. Each file carries
  `authorship:` frontmatter (`human`, `transcript`, or `agent`).
  **Authorship preservation is absolute**: never stylistically
  rewrite a human-authored file. Owner-directed *mechanical*
  transforms (a rename propagated everywhere) are fine and are logged.
- `wiki/` — the DM wiki, the single maintained master. Facts compiled
  from sources, interlinked, in encyclopedia tone. Holds the full
  table truth, DM secrets included.
- `dist/player/` — the player projection. Generated by
  `eddic project`; never edited by hand. Player-facing surfaces
  (published site, lore bot, player retrieval token) consume only
  this, by construction.

## Visibility — one truth, two audiences

Every wiki page carries `visibility:` frontmatter, and it **fails
closed**: no marker means DM-only. Only `visibility: player` projects.

For topics that reveal progressively, keep **twin pages**: the player
page under the canonical name (`characters/warden.md`, marked
`player`) holding exactly what the table knows now, and the DM page
adjacent with a `.dm.md` suffix (`characters/warden.dm.md`) holding
the full truth. The DM twin links its player twin; the player twin
NEVER links back — a player page may only link player pages. The
firewall (lint, and projection's refusal) enforces this mechanically.

Revealing is "lifting the veil": move newly-revealed facts into the
player twin (or flip a wholly-safe page's visibility), at the owner's
direction, and log it. When the DM asks for "a player's version" of a
DM page, write the twin from what the table has actually seen —
nothing the wiki can't attribute to play.

The catalogs are twins too: `index.md` is the player catalog (links
player pages only); `index.dm.md` is the DM catalog and lists
everything. Update both when pages are added or removed. Assets under
`wiki/assets/` project wholesale — never put a spoiler map there; DM
assets live under paths containing `.dm`, which never project.

## What to record, and in whose voice

Neutral, factual, encyclopedia tone. State settled facts; do not
interpret, analyze, or editorialize. If a detail has thematic weight,
state the detail and stop. When a fact is one character's account
(especially a self-serving one), attribute it — "by the Warden's
telling" — rather than stating it flat.

Write in a de-personalized voice: characters are figures in the
world, never tied to the people at the table. Published prose never
says "the player" or "the DM." Real names never enter the wiki.

Record at encyclopedia granularity: identities, relationships, world
rules, settled events, places, terms. Do not transcribe scenes beat
by beat; the sources hold moment-level detail.

## File format

Plain markdown. `#` page title, `##` sections. Connect pages with
relative `[text](path.md)` links — the links are the graph, and dense
linking is what lets an agent traverse it. Prefer plain prose; avoid
bold, lists, and tables unless genuinely required.

Every link must resolve. If you link a page that does not exist,
create the stub immediately: `#` title, any known facts, final line
reading `STUB`.

## Taxonomy

Inside `wiki/`: `characters/` (a directory with `overview.md` plus
topic files for principals; a single file for supporting figures;
`_unidentified.md` as the holding pen until a figure earns a file),
`places/`, `story/` (world rules and structural elements),
`concepts/` (named terms, oaths, institutions), `eras/`, `species/`,
`sessions/` (one recap per session), and `systems/` (table tooling,
not world lore). Add a category only when a page genuinely fits
nowhere; log it as `schema`.

## The operation log

`{{LOG}}` in the wiki directory is append-only. Each entry:
`## [YYYY-MM-DD] <type> | <summary>`, type one of `ingest` (sources
compiled into the wiki), `reconcile` (facts synced with external
tools, e.g. the timeline), `lint` (health-check pass), `schema`
(a change to conventions), `witness` (field notes reconciled from
the inbox). Convert relative dates to absolute. Never rewrite an
existing entry.

## Updating

Before creating a file, check the directory for a page that already
covers the topic; update rather than duplicate. When you update a
page, read it first and change only what changed — do not rewrite
surrounding prose to sound better (that is style, and style drift is
how wikis lose their voice). After any change: `eddic lint`, fix or
escalate per the lint pattern, re-`project`, and log.

Chronology belongs to the timeline tool (Ørlǫg) where the campaign
uses one; the wiki states time in plain prose and never invents a
date syntax. Reconcile only when the owner directs it, and log it.
